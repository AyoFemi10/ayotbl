Introduction

Most Telegram bot libraries make you choose between ease of use and type safety. ayotbl aims for both — and tries to stay current with Telegram's API without waiting on a library release cycle. Every method not yet given a typed wrapper still works through an escape hatch, so a brand-new Bot API method works the same day Telegram ships it.

This site documents every guide topic and walks through Bot API 7.0 through 10.2 version by version, with real, runnable code for each headline feature — not just a class-member dump.

Getting startedInstallation

terminal
npm install ayotbl

Requires Node.js 18+. ayotbl uses the native fetch, FormData, and Blob globals — there is nothing else to install.

Quick start

bot.ts
import { Bot } from "ayotbl";

const bot = new Bot(process.env.BOT_TOKEN!);

bot.command("start", (ctx) => ctx.reply(`Hey ${ctx.from?.first_name}!`));
bot.hears(/^ping$/i, (ctx) => ctx.reply("pong"));
bot.on("photo", (ctx) => ctx.reply("Nice photo!"));

bot.launch(); // long polling

Webhooks, one line

await bot.launchWebhook({ url: "https://your-domain.com", port: 8080 });
// or mount bot.webhookCallback('/webhook') into Express/Fastify/raw http yourself

GuideThree ways to write a handler

Pick whichever reads best for the moment. Mix freely — a decorator controller and a plain bot.command() call can coexist in the same bot, since decorators register onto the same underlying router.

1. Simple router

bot.command('start', ctx => ctx.reply('hi'));
bot.hears(/^ping$/i, ctx => ctx.reply('pong'));

2. Fluent builders

import { Keyboard, RichMessage } from "ayotbl";

bot.command("report", (ctx) =>
  ctx.replyRich(
    RichMessage.markdown()
      .text("Build report:")
      .table(["Step", "Status"], [["Install", "✅"], ["Deploy", "⏳"]])
      .checklist(["Install deps", "Run tests"])
      .build()
  )
);

3. Decorator classes

import { Bot, Context, Command, Hears } from "ayotbl";

class MainController {
  @Command("start")
  start(ctx: Context) { return ctx.reply("Welcome!"); }

  @Hears(/^thanks?$/i)
  thanks(ctx: Context) { return ctx.reply("You're welcome!"); }
}

bot.useController(new MainController());

Decorators use standard experimentalDecorators TypeScript output — a real tsc build (what npm consumers get) works correctly. If you run raw source through tsx for quick scripts, note that it doesn't honor that flag; compile with tsc first for decorator-based bots.

Keyboards

Inline keyboards attach buttons to the message itself. Reply keyboards replace the user's system keyboard. Both have fluent builders.

Inline keyboard, with styles and custom emoji

import { Keyboard } from "ayotbl";

const kb = Keyboard.inline()
  .button("Confirm", "confirm:yes", { style: "success" })
  .button("Delete", "confirm:delete", { style: "danger", icon_custom_emoji_id: "5368324170671202286" })
  .row()
  .url("Docs", "https://example.com")
  .copyText("Copy invite code", "AB12-CD34")
  .build();

await ctx.reply("Are you sure?", { reply_markup: kb });

Valid style values are "primary" | "success" | "danger".

Reply keyboard

const kb = Keyboard.reply()
  .text("Yes").text("No").row()
  .requestContact("Share contact")
  .build({ resize: true, oneTime: true, placeholder: "Choose one" });

Editing messages from button taps

The most common Telegram UI pattern — a user taps a button, the bot edits that exact message — works directly from inside a callback handler.

bot.action(/^confirm:(yes|no)$/, async (ctx) => {
  await ctx.answerCbQuery();
  await ctx.editText(ctx.match![1] === "yes" ? "Confirmed ✅" : "Cancelled ❌");
});

// ctx.editCaption(), ctx.editMedia(), ctx.editReplyMarkup() work the same way —
// all resolve the right chat_id/message_id whether you're inside a plain
// message handler or a callback_query handler.

Rich Messages Bot API 10.1 / 10.2

Structured, AI-assistant-style replies: headings, tables, checklists, blockquotes, and streamed drafts. Three ways to build one.

Markdown builder

import { RichMessage } from "ayotbl";

const rich = RichMessage.markdown()
  .heading("Build Report")
  .table(["Step", "Status"], [["Install", "✅"], ["Deploy", "⏳"]])
  .checklist(["Run tests", { label: "Deploy", done: false }])
  .build();

await ctx.replyRich(rich);

Structured blocks (Bot API 10.2)

For when you want guaranteed structure — real table colspan/rowspan — instead of round-tripping through markdown text.

const rich = RichMessage.blocks()
  .heading("Title", 2)
  .table([[{ text: "Header", is_header: true }], [{ text: "Cell" }]], { bordered: true })
  .details("Show raw logs", [{ type: "paragraph", text: "…" }])
  .build();

Streaming a draft while generating

// Ephemeral ~30s preview in a private chat — call sendRichMessage with the
// final content once generation finishes.
await bot.api.sendRichMessageDraft({
  chat_id: userId,
  draft_id: 1,
  rich_message: RichMessage.markdown().text("Thinking…").build(),
});

Multi-step conversations (Scenes)

import { Bot, Context, session, Stage, WizardScene } from "ayotbl";

bot.use(session());

const signup = new WizardScene("signup", [
  async (ctx) => { await ctx.reply("What's your name?"); await ctx.wizard.next(); },
  async (ctx) => { ctx.wizard.state.name = ctx.text; await ctx.reply("How old are you?"); await ctx.wizard.next(); },
  async (ctx) => { await ctx.reply(`Thanks, ${ctx.wizard.state.name}!`); await ctx.scene.leave(); },
]);

bot.use(new Stage([signup]).middleware());
bot.command("signup", (ctx) => ctx.scene.enter("signup"));

Each step runs once per incoming update while the scene is active. Call ctx.wizard.next() when ready to move on — it only advances the pointer, so the next step's body waits for the actual next message rather than firing immediately.

Sessions

import { session, RedisSessionStore } from "ayotbl";
import Redis from "ioredis";

bot.use(session({ store: new RedisSessionStore(new Redis(), { ttlSeconds: 86400 }) }));

bot.command("count", (ctx) => {
  ctx.session.count = (ctx.session.count ?? 0) + 1;
  return ctx.reply(`Count: ${ctx.session.count}`);
});

Defaults to MemorySessionStore (fine for local dev; state is lost on restart). Swap in RedisSessionStore for production — same interface, no other code changes.

Chat & role gates

Restrict handlers by chat type or sender role, without hand-rolling the check every time.

// Only respond to DMs
bot.privateChat(
  bot.command("start", (ctx) => ctx.reply("Welcome!"))
);

// Only in groups/supergroups
bot.groupChat((ctx, next) => next());

// Only chat admins or the creator (makes one getChatMember call per update)
bot.admin(
  bot.command("ban", (ctx) => ctx.reply("Banning…"))
);

// Only the creator
bot.creator(
  bot.command("shutdown", (ctx) => ctx.reply("Shutting down…"))
);

Composable with chatType('private' | 'group' | 'supergroup' | 'channel' | ChatType[]) for custom combinations.

i18n

import { createI18n } from "ayotbl";

const i18n = createI18n({
  defaultLocale: "en",
  locales: {
    en: { hi: (n: string) => `Hi ${n}!` },
    fr: { hi: (n: string) => `Salut ${n}!` },
  },
});
bot.use(i18n.middleware());
bot.command("start", (ctx) => ctx.reply(ctx.t("hi", ctx.from?.first_name)));

Locale is read from the user's Telegram language_code automatically, with a fallback to defaultLocale.

Flood control & throttling

429s are retried automatically everywhere — not just the polling loop.

const bot = new Bot(token, { floodControl: { maxRetries: 5 }, minIntervalMs: 35 });

Per-chat/per-user rate limiting for incoming updates, so one noisy user can't monopolize the bot:

import { rateLimit } from "ayotbl";
bot.use(rateLimit({ windowMs: 2000, limit: 3, onLimitExceeded: (ctx) => ctx.reply("Slow down!") }));

Serverless deployment

// Vercel
export default vercelWebhookHandler(bot);

// AWS Lambda
export const handler = lambdaWebhookHandler(bot);

// Cloudflare Workers
export default { fetch: cloudflareWebhookHandler(bot) };

Any method, typed or not

await bot.api.sendMessage({ chat_id, text: "typed and autocompleted" });

// Works the same day Telegram ships a new method
await bot.api.call("someBrandNewMethod", { chat_id, whatever: true });

Every method not yet given a typed wrapper still works — you're never blocked waiting on a library release.

Bot API, version by version7.0 – 7.11

The foundation of the modern Bot API: structured replies, reactions, boosts, giveaways.

7.0ReplyParameters & message reactions

Replaced the old flat reply_to_message_id with a structured object that can quote part of a message, reply across chats, or react with an emoji.

bot.on("text", async (ctx) => {
  await ctx.api.setMessageReaction({
    chat_id: ctx.chatId!,
    message_id: ctx.message!.message_id,
    reaction: [{ type: "emoji", emoji: "👍" }],
  });

  await ctx.reply("Got it, quoting your message:", {
    reply_parameters: {
      message_id: ctx.message!.message_id,
      quote: ctx.text?.slice(0, 30),
    },
  });
});
7.0Chat boosts & giveaways
bot.on("chat_boost", (ctx) => {
  const boost = ctx.update.chat_boost!.boost;
  return ctx.reply(`New boost, expires ${new Date(boost.expiration_date * 1000).toDateString()}`);
});

bot.on("message", (ctx) => {
  if (ctx.message?.giveaway_completed) {
    return ctx.reply(`Giveaway ended — ${ctx.message.giveaway_completed.winner_count} winners`);
  }
});
7.11Copy-to-clipboard buttons
Keyboard.inline().copyText("Copy invite code", "AB12-CD34")

8.0 – 8.3

Gifts and Telegram Stars arrive.

8.0Sending a gift
const { gifts } = await bot.api.getAvailableGifts();

await bot.api.sendGift({
  gift_id: gifts[0].id,
  user_id: ctx.from!.id,
  text: "Enjoy! 🎁",
});
8.0Stars subscription invoice
const link = await bot.api.createInvoiceLink({
  title: "Premium access",
  description: "Unlock premium features",
  payload: "premium_sub",
  currency: "XTR",
  prices: [{ label: "Monthly", amount: 100 }],
  subscription_period: 2592000, // 30 days, required for XTR subscriptions
});
8.2Verifying an account
await bot.api.verifyUser({ user_id: ctx.from!.id, custom_description: "Verified via ayotbl demo" });

9.0 – 9.6

Business accounts, Checklists, Suggested Posts, Managed Bots, and a reworked poll model.

9.0Responding to a business message

Every business message carries business_connection_id — required to call any Business Account method in response.

bot.on("business_message", async (ctx) => {
  const connectionId = ctx.message?.business_connection_id;
  if (!connectionId) return;

  await bot.api.setBusinessAccountBio({
    business_connection_id: connectionId,
    bio: "Managed by ayotbl",
  });
});
9.1Checklists
await bot.api.sendChecklist({
  business_connection_id: connectionId,
  chat_id: ctx.chatId!,
  checklist: {
    title: "Launch checklist",
    tasks: [
      { id: 1, text: "Write tests" },
      { id: 2, text: "Update docs" },
      { id: 3, text: "Publish" },
    ],
  },
});
9.2Suggested Posts (Direct Messages in channels)
await ctx.reply("Proposed post text", {
  suggested_post_parameters: {
    price: { currency: "XTR", amount: 50 },
  },
});

bot.on("message", async (ctx) => {
  if (ctx.message?.suggested_post_info?.state === "pending") {
    await bot.api.approveSuggestedPost({ chat_id: ctx.chatId!, message_id: ctx.message.message_id });
  }
});
9.6Multi-answer quiz polls

correct_option_id (singular) was replaced by correct_option_ids — a quiz can now have more than one correct answer.

await bot.api.sendQuiz({
  chat_id: ctx.chatId!,
  question: "Pick every even number",
  options: ["1", "2", "3", "4"],
  correct_option_ids: [1, 3],
});

10.0 – 10.2

Guest mode, Rich Messages, Ephemeral Messages, and Communities — the newest additions, verified directly against the official changelog the same week they shipped.

10.0Reacting and un-reacting programmatically
await bot.api.deleteMessageReaction({ chat_id: ctx.chatId!, message_id, user_id: ctx.from!.id });
await bot.api.deleteAllMessageReactions({ chat_id: ctx.chatId!, message_id });
10.1Rich Messages

See the Rich Messages guide above for the full builder API.

await ctx.replyRich(RichMessage.markdown().heading("Status").text("All systems normal.").build());
10.2Ephemeral Messages

A reply visible only to the user who triggered it — useful for admin panels or private confirmations in a group.

await bot.api.sendMessage({
  chat_id: ctx.chatId!,
  text: "Only you can see this ↴",
  receiver_user_id: ctx.from!.id,
});

// Later, edit just that ephemeral reply:
await bot.api.editEphemeralMessageText({
  receiver_user_id: ctx.from!.id,
  chat_id: ctx.chatId!,
  ephemeral_message_id: 1,
  text: "Updated, still only visible to you.",
});
10.2Communities
bot.on("message", (ctx) => {
  if (ctx.message?.community_chat_added) {
    return ctx.reply(`Added to community: ${ctx.message.community_chat_added.community.title}`);
  }
});

Status & testing

Verified: full build pipeline (tsc → 68 tests → JS smoke test → real-decorator smoke test). Core method coverage diffed against telegraf's actual method list. Live-tested against real Telegram servers for the core happy path — commands, inline keyboards, callback queries, text matching, photo receipt.

Not yet verified: webhook mode, Scenes in a live multi-turn conversation, file uploads, and the newer feature areas (Rich Messages, Ephemeral Messages, Gifts, Stars) — all unit-tested against mocked fetch, never a real server.

No production mileage yet. That only comes from real usage over time — it's not something more code can fix ahead of time.
